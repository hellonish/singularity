# report_generator

**Tier**: output  
**Name**: `report_generator`  

## Description

The `report_generator` skill is the terminal synthesis and composition engine for producing a formal, structured research report. It physically performs the final cognitive integration of all upstream research artifacts, raw data, and analytical insights into a cohesive, audience-tailored written document. The process involves a multi-stage transformation: First, it ingests a comprehensive collection of data points, source summaries, extracted quotes, and analysis conclusions from prior nodes in the execution graph. Second, it applies a sophisticated cognitive framework to organize this heterogeneous input into a logical narrative flow, adhering to academic and professional reporting standards. Third, it executes a detailed writing process, generating original prose for each mandated section, synthesizing information, creating an executive summary that encapsulates core findings, and meticulously weaving inline citations into the narrative fabric. The final output is a polished document that not only presents findings but contextualizes them within the stated methodology, acknowledges investigative limitations, and provides a traceable reference backbone. This is a high-stakes, high-complexity transformation from fragmented evidence to authoritative communication.

## When to Use

- **Primary Scenario**: This skill MUST be used as the final, terminal output node in an execution plan when the plan's metadata explicitly specifies `output_format: 'report'`. It is the capstone of a research pipeline.
- **Upstream Dependencies**: This skill REQUIRES robust, processed input from upstream nodes. It expects a consolidated body of evidence including, but not limited to:
    - **Source Summaries**: Concise syntheses of individual sources (e.g., from `source_summarizer`).
    - **Extracted Data & Quotes**: Key facts, statistics, and direct quotations harvested from sources (e.g., from `data_extractor` or `quote_collector`).
    - **Analytical Conclusions**: Insights, comparisons, or answers generated by analysis nodes (e.g., from `analyst` or `comparison_engine`).
    - **Citation Metadata**: Complete bibliographic data for all referenced sources (Author, Year, Title, etc.), typically aggregated by a prior node.
    The input should be a rich, structured context that the LLM can query to populate all report sections.
- **Downstream Nodes**: Typically, NO nodes follow `report_generator` as it is the planned output. Its output (`OutputDocument`) may be passed to a rendering or delivery system (e.g., a PDF formatter, email sender, or UI presenter), but those are not part of the cognitive execution DAG.
- **Edge Cases - DO NOT USE**:
    - When the requested output is a brief answer, summary, list, or data table (`output_format` is 'answer', 'summary', 'list', 'table'). Use a simpler, more focused skill.
    - When upstream data is extremely sparse or of poor quality; the generator cannot hallucinate credible content and will produce a low-quality or misleading report.
    - As an intermediate step in a larger analysis; it is designed for final presentation.
    - If the `output_format` metadata is missing or ambiguous; the planner must resolve this first.

## Execution Model

LLM-based

**Prompt file**: `prompts/report_generator.md`

## Output Contract

OutputDocument — sections: exec_summary, methodology, findings, limitations, references

## Constraints

- **Length Targets Are Strict**: The generated report must adhere precisely to the following word count targets based on the specified audience persona. The Planner must provide the correct audience context. These are not suggestions but requirements:
    - **layperson**: ~600 words
    - **student**: ~1500 words
    - **practitioner**: ~2000 words
    - **expert**: ~3000 words
    - **executive**: ~400 words
- **Mandatory Disclosure of Gaps**: The report **MUST** explicitly disclose, within the 'limitations' or 'methodology' section, all PARTIAL or FAILED coverage from upstream nodes. If a node failed to retrieve information, provided incomplete data, or could only access a subset of intended sources, this gap in evidence **must be stated clearly**. Do not obscure missing information.
- **Citation Format is Inviolable**: All references to sources **MUST** use the inline citation format `[Author YYYY]` (e.g., `[Smith 2023]`). This format must be used consistently throughout the 'findings' and 'exec_summary' sections. Do not use numeric citations, footnotes, or hyperlinks.
- **No Hallucination of Content**: The report must be grounded **exclusively** in the provided context from upstream nodes. Do not invent facts, figures, conclusions, or sources. If the context lacks information for a required section, state that clearly rather than fabricating content.
- **Structural Compliance**: The output **MUST** contain the five exact sections specified in the Output Contract: `exec_summary`, `methodology`, `findings`, `limitations`, `references`. Do not add, remove, or rename these sections.
- **Audience Appropriateness**: Tone, depth, and jargon must be tailored to the specified audience (e.g., avoid technical jargon for 'layperson', assume domain knowledge for 'expert').
- **Token Limit Awareness**: Generating a 3000-word report is a significant LLM context window operation. The Planner must ensure the *combined* input context (upstream data + instructions) does not exceed the model's context limit, potentially requiring intelligent summarization of upstream data before passing it to this skill.
